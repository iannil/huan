## Phase 3：Port Hugo WordCount 算法（核心突破 + 规范纠错）

### 关键发现：原规范对 Hugo 算法的描述是错的

规范说「Hugo 的 WordCount 用 `unicode.Is(unicode.Han, r)`」，每条 Han 字符算 1 词。**这是错的。** 实际读 `hugolib/page__content.go`（Hugo master）发现，Hugo 的 CJK WordCount 是：

```go
result.plain = StripHTML(rendered.content)
result.plainWords = strings.Fields(result.plain)
if isCJKLanguage {
    for _, word := range result.plainWords {
        runeCount := utf8.RuneCountInString(word)
        if len(word) == runeCount {  // 纯 ASCII
            result.wordCount++
        } else {                     // 含多字节（CJK 等）
            result.wordCount += runeCount  // 每个 rune 都算
        }
    }
}
```

也就是说：先按空白拆字段（`strings.Fields` 用 `unicode.IsSpace`，含表意空格 U+3000）；每个字段若是纯 ASCII 算 1 词，否则算它的 rune 总数。**CJK 字段内的标点也按 rune 计入**，因为它们没有空白分隔。

### 验证（zhurongshuo books/index.html）

| 书 | huan 修复前 | huan 修复后（int div） | Hugo |
|---|---|---|---|
| reality-construction | 12.0 万 | 15.0 万 | 15.5 万 |
| advancement-of-reality | 16.0 万 | 16.0 万 | 16.7 万 |
| the-observers | 8.0 万 | 12.0 万 | 12.1 万 |
| god-beyond-observation | 13.0 万 | 19.0 万 | 19.6 万 |

修复前字数严重偏低（最多差 50%）。修复后字数与 Hugo **完全一致**（如 the-observers：huan 算出 121416，Hugo 也算出 121416），显示的整数部分也匹配，只是小数部分还差一点点 —— 这是 `div` 函数的另一个 bug（见下）。

### 新发现的下一个 bug：`div` 函数只支持 int

模板 `约 {{ lang.FormatNumberCustom 1 (div $totalWords 10000.0) }} 万字`，其中 `10000.0` 是浮点字面量。但 huan 的 `div` 签名是 `func(a, b int) int`，所以 `121416 / 10000.0` 在 huan 里被强转成 int 除法 = 12，显示 "12.0"；Hugo 的 `div` 支持浮点，得 12.1，显示 "12.1"。

### Hugo 真实 tiebreaker（逆向出来的）

日期相等时按以下顺序排：
1. `DateParsed` desc（已经是这层）

3. `RelPath` asc

### 设计观察

- `commonPrefixLen` 的关键作用：`TruncateHTMLByWords` 返回的 truncated 串可能在切点追加了合成 close tag（如 `</strong></p>`），这些 byte 不在原 HTML 对应位置——所以**最长公共前缀**正好是真实切点（truncated 与原 HTML 的第一个分歧 byte）。这个 trick 让我们能精确还原切点在原 HTML 中的 byte offset，不需要重构 `TruncateHTMLByWords` 暴露内部状态。
- 26 种 block close tag 的清单来自 HTML 规范的 block-level elements（加 `<figure>` / `<figcaption>`）。Hugo 实际算法（GitHub issue #11863）没有明确文档化完整清单，是从 HTML spec 推断 + 覆盖 zhurongshuo 实际内容的所有 block 类型。
- `TruncateHTMLToBlockBoundary` 设计为 `TruncateHTMLByWords` 的**包装器**，不改原函数——保持 Phase 5 的 word-boundary 行为纯净，本 phase 只调 caller 层策略。
