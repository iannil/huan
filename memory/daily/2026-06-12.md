## grill-with-docs：RSS items 顺序 tiebreaker（已 rollback）

用户用 `/grill-with-docs` skill 想做 stage 1 剩余 5 类差异里的 #2「RSS items 顺序 tiebreaker」。

### Grill 收敛结论（保留作未来起点）

1. **CURRENT_STATE.md 描述错误**：第 2 类差异"Hugo date 相同时顺序不稳定"——实证发现 Hugo **是稳定**的，有确定性 tiebreaker
2. **Hugo 的 tiebreaker 规则**（逆向）：观察 home RSS 前 15 项是「第10→15→1→2」，匹配**阿拉伯数字字典序**。推断规则是 `Date desc → lower(Title) asc → RelPath asc`（zhurongshuo 无 linkTitle，LinkTitle 自动 fallback 到 Title）
3. **huan 当前问题**：`internal/content/tree.go:311` 的 `sortPagesByDateDesc` 是插入排序，只按 Date 比较，date 相同时保持文件扫描顺序——不对齐 Hugo
4. **设计选择**：在 content 层 `sortPagesByDateDesc` 加多字段 tiebreaker（影响所有默认排序遍历，理论上 Hugo 那边也是这套规则，应该一次性对齐）+ 加单元测试覆盖 Date desc / Title asc / Path asc

### 撞墙过程

1. ✅ 实现 sortPagesByDateDesc tiebreaker（Date desc → lower(Title) asc → RelPath asc，改用 sort.SliceStable）
2. ✅ 写 5 个单元测试（Date / Title / case insensitive / Path / shuffle stability）
3. ✅ go test 全过
4. ❌ diff 验证：原本期望 diff 减少，但实际从 937 → 908（净增 29 newly diffed），新差异都在 `books/volume-X/.../index.html`
5. 🔍 调查发现：sortFunc 在 `internal/template/funcs.go:556` 有独立 bug——无 field 参数时**不排序**（缺 else 分支）。模板 `{{ $partSlugs := sort (...) }}`（无 field）因此 partSlugs 顺序依赖 RegularPagesRecursive 的扫描顺序，我的 sortPagesByDateDesc 改动通过这个 bug 传染到了 books list
6. ✅ 修了 sortFunc，但 diff 没退回 baseline
7. 🔍 深入调试：实证发现 sort 的**返回值根本没被用**——把 sort 改成返回反序、改成 in-place mutate input slice（PageSlice 和 []interface{} 两个分支），HTML 都不变
8. 🧱 撞墙：Go template + Scratch + `:=` 在 pipe + variadic + 多返回值场景下有深层引用语义问题，HTML 实际遍历的是 `$scratch.data[key]` 的原始 slice，不是 sort 返回值。无法在合理时间内解决
9. 📉 按 CLAUDE.md「错误数异常上升立即回滚」规则，**全 rollback**（funcs.go、tree.go、tree_test.go）
