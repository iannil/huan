### 经验教训

1. **broken baseline 隐形破坏所有数字**：本地 layout 截断 → 96% 页面无 body → huan 与 Hugo 都无 body → "假性相同"。grill-me 之前报告的"stage 2 完成"全是基于 broken baseline。**任何对比工作开始前必须实证 baseline 自身正确**（对比线上 / HEAD / 已知 good 状态）。

3. **plainify 必须返回 template.HTML**：Hugo 的 plainify 也是这么干的（返回 safe HTML）。否则 Go template 会 auto-escape 把 `&quot;` 里的 `&` 转 `&amp;`，导致 description meta 出现 `&amp;quot;`（双逃逸）。tdewolff minify 之后没法把双逃逸还原回 ASCII `"`。
4. **canonify 必须区分代码块内外**：`href=/api/foo` 在真 HTML attribute 里要 canonify，但在 `<code>` 里是代码示例的"字符串"，canonify 它就破坏了示例。用 regex split 把 `<code>...</code>`/`<pre>...</pre>` 区段 skip。

### 关键技术发现

1. **blake3 hash 算法**（Step 4）：通过反向工程 wrangler 源码 + python blake3 参照向量，确认 `hash = blake3(base64(content) + ext).hex()[:32]`。**不是 SHA256**（ADR 0002 §7 原文有误，已在 §0 修订中改写）。任何字节错误都会让 dedup 全失效。
2. **JWT 双 token 模型**（Step 5）：`apiToken` 用于 upload-token GET + deployment POST；`jwt`（300s 短期）用于 assets/*。JWT refresh 通过 `Client.InvalidateJWT + 单次 retry` 实现。
3. **5 步协议不是 4 步**（Step 6）：Plan agent 验证发现 ADR 原画的「check-missing → upload → deployment」漏了 `upsert-hashes` 步骤（告诉 CF 本次部署的完整 hash 列表，不只是新传的）。已修订。
4. **HTTP 并发硬上限 3**（Step 5/6）：与 `--concurrency`（管 hash 并行）解耦。HTTP POST 并行超过 3 撞 CF gateway 5xx。3 个 batch 并发上传，遇 5xx 走 retry。
5. **`goto` 不能跳进 block**（Step 7）：plugin.go 初版用 goto unsupported，Go 编译器拒绝「jump into block」。改成显式 `hasUnsupportedTarget()` helper。
6. **回引号字符串不能内嵌回引号**（Step 6）：`len(`name=""`)` 在 Go 中会被解析成「len + ( + 字符串 + )」但字符串内的 `"` 让 parser 困惑，需要 `"name=\""`。
7. **测试 backoff 必须 fast**（Step 5）：默认 backoff 200ms/1s/5s 让 4 个 retry 测试累计 ~14s。`fastBackoff(t)` helper 用 t.Cleanup 还原，让 retry 测试 ~1s。
8. **mutilpart 解析 mock 简化**（Step 6）：手写 boundary split + `name="..."` 抽取，避免引入 mime/multipart parser 的依赖。验证 manifest 前 `/` + branch 透传。

### 经验教训

1. **plan agent 的研究价值**：Plan agent 反向工程 wrangler 源码发现了 4 处 ADR 0002 错误（SHA256→blake3、4→5 步、单 token→双 token、并发模型）。如果直接按 ADR 实施，整个 dedup 会失效。**含义**：任何「按文档/规范实施 API 协议」前，必须先验证协议（参考实现 > 文档）。

Commit `8725935`：`feat(cloudflare): add Worker modules upload (PR3)`，+946 / -16。

### 关键技术发现

1. **CF Workers multipart 必须有 filename header** —— script part 的 Content-Disposition 必须是 `form-data; name="<basename>"; filename="<basename>"`，且 part name 与 filename 一致（都是 script basename）。缺 filename CF 拒绝。原 PostForm 只支持 name，必须扩展 formField struct + 加 PutForm 方法。
2. **binding 类型映射 CF 期望的 JSON 字段名**：
   - r2_bucket → `bucket`
   - kv_namespace → `namespace_id`（snake_case）
   - d1 → `id`
   - vars/secret_text → `text`（不是 `value`）
   
   用户在 yaml 用 `bucket/namespaceId/id/value`（camelCase），代码映射到 CF 期望的 snake_case / 不同字段名。这是 ADR 0002 §8 「yaml 全量声明」的具体含义。
3. **route 同时填 `zone` 和 `zone_name`** —— CF API 不同版本期望不同字段。zone 名相同时两边都填保证向前向后兼容。
4. **CompatibilityDate 默认 "2024-01-01"** —— 与 wrangler 一致；用户可在 yaml 覆盖。
5. **WorkerConfig.CompatibilityDate 与生产 deploy 解耦** —— validate 不要求该字段（因为 deploy 时有默认值）。plugin info 反映用户写的值（可能空），实际 deploy 时填充默认。

### 关键技术发现

1. **C4 是误判** —— audit agent 1 说「wrangler 分片 check-missing 到 2000 hashes/chunk」，实证 wrangler `assets/check-missing` 一次性发所有 hashes（20k 站点 ~640KB，CF 接受）。撤回该修复。

2. **H6 也基本是误判** —— audit 担心 unknown binding type 会污染 JSON（r2_bucket 上同时出现 namespace_id）。实证：switch case 只 set 对应字段；零值字段由 `omitempty` 自动跳过。无 bug。

3. **H3 prune scope 不是 bug** —— audit 担心「user mapping `to: images/gallery`，R2 有 unrelated `images/banners/x.jpg`，会被 prune」。实证 `listAll` 只列配置的 prefix，前缀外的对象 prune 看不到。代码本身正确，缺的是 observability（per-delete 日志）+ 文档。

4. **真 bug 集中在 concurrency 模型** —— `--concurrency` flag 从 CLI 读了但永不传到 deployer；defaultConcurrency() 函数定义了从不调用；gateway 5xx 自动降级完全未实施。这三个一起修。

5. **C1 log.SetOutput(io.Discard) 是隐性污染** —— minio-go 用 stdlib log，huan 其它地方（build/serve）也用，全局静默会吃掉所有人的日志。

6. **L1 trace ID 空** —— `newTraceIDForDeploy()` 返回空字符串再让 logger 生成，导致 early-return Report.TraceID 是空。修：导出 `deploy.NewTraceID()`，提前生成。

### 经验教训

1. **审计 agent 报告需要实证复核** —— agent 1 的「C4 wrangler 分片 check-missing」与 agent 2 的「H6 binding 字段污染」都是基于推测，实证后撤回。**含义**：agent 报告的「事实声明」（如「wrangler uses 2000-hash chunks」）必须 grep wrangler 源码或跑 demo 验证；不能直接信任。

2. **跨 PR 的「未实施」断言会反向失效** —— PR2 的 `TestR2Sync_PluginDispatch_WorkerTarget` 断言「worker PR3 not yet」；PR3 实施后必须主动改回。**含义**：跨 PR stub 测试要主动维护，PR 实施后立刻 grep 找反向失效的断言。

3. **concurrency 模型必须在第一 PR 就画清楚** —— PR1 实施时硬编码 `parallel := 3` 看似简单，但留下了 C2（flag 不生效）+ H2（R2 混淆 CPU/HTTP）+ H1（无降级）三个坑。审计时一次发现。**含义**：涉及并发的代码，必须在实施时就明确「CPU 并行 vs HTTP 并行」「默认值」「失败时的降级策略」三个维度，写进 ADR。

4. **stdio 全局 mutation 是隐形 bug 源** —— `log.SetOutput(io.Discard)` 在 NewR2Syncer 里看似无害（只静默 minio 的噪音），实际吃掉了整个进程的 stdlib log。**含义**：构造器里绝不修改全局状态（log / env / cwd）；要么局部用，要么完全不依赖。

5. **observability 是「代码正确」的必要补强** —— H3（prune scope）代码本身正确，但用户事后无法审计删了什么。加 per-delete 结构化日志让代码「可证明」正确。**含义**：destructive 操作（delete、overwrite）必须每条都日志；只输出 aggregate count 不够。

6. **dryRunReport 语义陷阱** —— `Skipped: len(assets)` 字面意思是「跳过」，但 dry-run 没查过 remote，不知道哪些会真跳过。**含义**：Report 字段语义必须明确（Attempted = tried；Succeeded = uploaded；Skipped = remote-already-had；Failed = exhausted retries）；dry-run 时不知道的字段一律 0，不要瞎填。

### 关键经验

- **BSD sed `\b` 不工作**：macOS 默认 sed 不支持 GNU 的 word boundary，`s/deploy\.NewLogger\b/.../g` 会漏掉 `deploy.NewLogger(traceID)`。用 `deploy.NewLogger(` 加左括号作 disambiguator。
- **//go:embed 文件要 trim**：VERSION 文件用文本编辑器保存带尾随 `\n`，embed 后字符串也带 `\n`，下游 Printf 输出多一行。在 init() 里 `strings.TrimSpace` 即可。
- **atomicWrite 必须用 sibling temp**：`os.CreateTemp(dir, ".tmp-*")` 必须用 final 的目录，跨文件系统 rename 失败。这就是"原子写"的标准模式。
- **observability 提取不是 premature abstraction**：Logger 当前已 domain-agnostic，住 deploy 包是历史偶然。Rule of Three 针对"为想象中的复用创造抽象"，而这里抽象已存在，只是包名错了——纯组织性 refactor。
