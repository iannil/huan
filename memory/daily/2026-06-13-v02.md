## v0.2.0 — remove encrypt（commit `5c220e2`）

### 任务

v0.1.0 发版后审视代码发现 `internal/encrypt/` + `internal/shortcode/redact.go` 是 zhurongshuo 历史遗留功能：
- zhurongshuo 实际**未启用**加密（content/ 里没有 `access: protected` frontmatter）
- 加密密文生成本来就由外部 Node.js `scripts/encrypt-content.js` 完成，huan 只读不写
- 维护这套未启用的代码 = 持续负担（每次重构都要绕开它）

**决策**：彻底移除，不保留 dead code。`huan.yaml` 的 `params.encryptGroups` 留作向后兼容（不删配置，但 huan 不再消费）。

## v0.2.1 — toc/export/sync subcommands + multi-archetype new（commit `afe89a9`）

### 任务

zhurongshuo Hugo→huan 迁移 Phase 1：把 zhurongshuo 项目里依赖 Node.js / bash 编排的运维命令，port 成 huan 原生子命令，让 zhurongshuo 能完全脱离 Node 工具链。

### 4 个子命令

| 命令 | 对标 | 输出契约 |
|---|---|---|
| `huan new <kind>/<path>` | Hugo archetypes（增强） | 顶部 path segment 选 `archetypes/<kind>.md`，回退 `default.md`，最后回退内置 |
| `huan sync gallery` | 自定义 shell 脚本 | 为 `static/images/gallery/` 新图自动 scaffold `content/gallery/<name>.md`（R2 上传独立走 `deploy cloudflare r2`） |
| `huan toc` | `scripts/generate-toc.js` | byte-identical 输出到 `developer/toc/{books,practices,products}-toc.md` |
| `huan export` | `scripts/export.sh` | md5-identical CSV 到 `developer/祝融说_副本<date>.csv`，用 `i18n.BuildCollator` 复现 zh_CN.UTF-8 排序 |

### 改动（净 +1014/-11 行）

| 文件 | 内容 |
|---|---|
| `cmd/huan/export.go` | +191，CSV 导出（frontmatter → CSV 行 + i18n 排序） |
| `cmd/huan/sync.go` | +137，gallery 扫描 + scaffold |
| `cmd/huan/toc.go` | +365，TOC 生成（content tree 遍历 + markdown 输出） |
| `cmd/huan/newcmd.go` | +60，multi-archetype 路径解析 |
| `cmd/huan/main.go` | +2/-1，注册新子命令 |
| `cmd/huan/{export,sync,toc,newcmd}_test.go` | +485 测试 |
| `internal/version/VERSION` | 0.1.0 → 0.2.0 |

### 关键技术点

- **`huan toc` byte-identical**：原 `scripts/generate-toc.js` 用 Node fs + 特定字符串拼接顺序。Go port 必须逐字节匹配（包括缩进 / 空行 / 标点）。测试用 golden file 对比。
- **`huan export` md5-identical**：CSV 行的排序是关键——`scripts/export.sh` 用 `LC_ALL=zh_CN.UTF-8 sort`，Go port 用 `i18n.BuildCollator("zh-cn")`（stage 2 phase 2 已经引入的 collator）保证拼音序一致。
- **`huan new` 多 archetype**：Hugo 的 archetype 是 `archetypes/default.md` 单模板。zhurongshuo 实际需要按内容类型分（post / book / practice / product），所以扩展成 `archetypes/<kind>.md` 多模板，顶部 path segment 选 kind。

### 验证

- `go test ./...` 全 PASS
- 手动在 zhurongshuo 跑 `huan toc` / `huan export` 对比 `scripts/generate-toc.js` / `scripts/export.sh` 输出，byte/md5 一致
- `huan new post/test-post` 用 `archetypes/post.md`，`huan new books/volume-4` 用 `archetypes/books.md`

### 教训

- **port 脚本时要复用已有基础设施**：`huan export` 没有重新实现 zh_CN 排序，直接用 `i18n.BuildCollator`（stage 2 phase 2 投入）。这是「Investment in Collator pays off」的实例。
- **byte-identical 是硬约束**：用户已经在用原脚本的输出做下游处理（git diff / diff 工具）。Go port 输出任何字节差都会让 git 显示"全文件 diff"，破坏 review 流程。
- **kind-specific archetype 是 zhurongshuo 实际需求驱动的**：Hugo 默认 single-archetype 不够，所以扩展。但接口保持向后兼容（无 kind 时回退 default.md）。

## v0.2.2 — CI auto-create GitHub Release on tag push（commit `e3dd9f3`）

### 任务

v0.2.1 之前发版流程：本地 `go run ./cmd/huan release` → 手动 `gh release create v0.2.x release/0.2.x/*` → 手动上传。痛点：
- 容易忘上传某个 tarball
- checksums.txt 与 GitHub Release assets 是两套校验源
- 不能从 tag 一键得到可下载产物

### 实施（.github/workflows/release.yml，81 行）

| 步骤 | 内容 |
|---|---|
| Trigger | `push.tags: ["v*"]` + `workflow_dispatch` (input: tag) |
| Permissions | `contents: write`（gh release create 需要） |
| Resolve tag | env 传递 + grep `^v[0-9]+\.[0-9]+\.[0-9]+$` 格式校验（防 injection） |
| Checkout | `ref: ${{ steps.tag.outputs.tag }}`（pin 到 tag，不用默认 HEAD） |
| Setup Go | 1.23，`cache: false`（避免 cache poisoning） |
| Build | `go run ./cmd/huan release`（dogfooding，规范调用） |
| Read VERSION | `tr -d '\n' < internal/version/VERSION` + 格式校验 |
| Create Release | `gh release create "$TAG" release/$VERSION/*` 或 `gh release upload --clobber`（re-run 幂等） |

### 安全考量

- **GitHub Actions injection**：tag name 经过 `grep -Eq '^v[0-9]+\.[0-9]+\.[0-9]+$'` 严格校验后才用于 `ref:` 和 `run:`。workflow_dispatch 的 input 同样校验。
- **Cache poisoning**：`cache: false` 避免 setup-go 缓存被恶意 PR 投毒。
- **Tag vs VERSION 一致性**：workflow 用 tag（外部）和 VERSION（内部）双源，任何一个格式错都 fail-fast。

### Back-fill 支持

`workflow_dispatch` + tag input 让我们可以为「workflow 之前已 push 的 tag」补建 Release：
- v0.1.0 / v0.2.0 / v0.2.1 三个 tag 在 workflow 落地前就 push 了
- v0.2.2 是首个走自动流程的 tag
- 之后用 workflow_dispatch 手动给历史 tag 补 Release

### 验证

- v0.2.2 tag push 后自动跑通，GitHub Release v0.2.2 含 5 tarball + checksums.txt + manifest.json
- `gh release view v0.2.2` 显示所有 assets
- 测试 re-run 幂等：第二次 push 同 tag 走 `gh release upload --clobber`，不报错

### 教训

- **CI workflow 是发版工具的最后一公里**：`huan release` 只产本地 `/release/{version}/`，但要变成用户可下载的 GitHub Release assets，需要 CI 把产物 upload。两者职责清晰分离。
- **dogfooding 是规范的一部分**：CI 里用 `go run ./cmd/huan release` 而不是预编译 binary，与 ADR 0004 §Q14 一致——operator 透明，无预构建步骤。
- **幂等是发版 workflow 的硬要求**：re-run 不能报错（gh release create 已存在时改 upload + clobber），否则任何 CI flake 都需要人工干预。
