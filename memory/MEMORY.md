## 关键决策

- 模板引擎阶段一用 `html/template`，阶段二可插件替换
- Markdown 用 goldmark（与 Hugo 同源库）
- 配置格式 `huan.yaml`（YAML），非 drop-in 替换 Hugo
- 验证方式：`./scripts/diff-build.sh` 多模式对比（byte 雷达 + normalized / seo / ai 三维度门禁），三维度任一失败则阻断合并
- 插件架构：**统一插件系统**（[ADR 0003](../docs/adr/0003-unified-plugin-system.md)）—— `internal/plugin/` 顶层 host（Plugin{Name()} + Registry + Find[T]），capability 接口分散在领域包（首个 `internal/deploy/types.go::Deployer`），yaml 顶层 `plugins:<name>.*`，`${VAR}` strict 插值，CLI `huan deploy <name>` + `huan plugin list/info`，编译期 hardcoded 注册在 `cmd/huan/plugins.go`。**首期只画 `Deployer`，不预画 `PaymentProvider` 等（YAGNI）**。Cloudflare deploy 插件是首个实例（[ADR 0002](../docs/adr/0002-cloudflare-deploy-plugin.md)）。**三个 target 全部实施完毕**：pages（PR1，blake3+5-endpoint+dual-token）/ r2（PR2，minio-go+MD5 etag+prune）/ worker（PR3，multipart modules API）。13 commits / ~7300 行 / ~140 测试 PASS 含 `-race`
- **v0.2.0 移除 `internal/encrypt/` + `shortcode/redact.go`**（zhurongshuo 未启用，详见 [ADR 0005](../docs/adr/0005-remove-encrypt-and-v02-feature-batch.md)）。**v0.2.3 进一步移除 `huan.yaml` 的 `params.encryptGroups` dead config**（反转 ADR 0005 §1.2，详见 [ADR 0006](../docs/adr/0006-remove-encryptgroups-dead-config.md)）—— zhurongshuo 仓库的 yaml 已先行清理，huan 仓库本次跟上。加密密文生成器 `scripts/encrypt-content.js` 不存在（huan 仓库无 / zhurongshuo 仓库也无）
- serve 模式用临时目录 `os.MkdirTemp("", "huan-serve-*")`，绝不污染 `docs/` 生产输出
- rebuild 用原子 swap（sibling staging dir + rename），保证 rebuild 期间无 404
- `BuildSite` 非并发安全，rebuild 通过 `atomic.Bool` 串行化 + pending 合并
- **三维度等价标准（2026-06-12，ADR 0001）**：stage 1 验收从「逐字节 100% 一致」改为「肉眼 / SEO / AI 三维度与 Hugo 输出无差异（甚至更好）」。byte-diff 保留作回归雷达，三维度对比作为门禁。允许修正型 + 扩展型「更好」（不破坏基线即可）。
- **Cloudflare 发布插件设计（2026-06-13，ADR 0002）**：全量 Cloudflare 化（HTML→Pages / 图片→R2 / image-resizer→Worker）做成 huan 内置插件。**纯 Go 直连 Cloudflare API**（不 shell out wrangler，不引 Node.js），R2 走 minio-go，配置在 `plugins:<name>.*` + `${VAR}` strict 插值（环境变量）。CLI 三层 `huan deploy cloudflare [pages|r2|worker]` + `huan plugin list/info`。`internal/deploy/types.go` 定义 `Deployer` capability 接口（嵌入 `plugin.Plugin`），Registry 在 `internal/plugin/`。**仅设计文档落地，未实施**——首 PR 范围 = plugin 系统 + Cloudflare Pages only（R2/Worker 后续 PR）。
- **统一插件系统设计（2026-06-13 晚，ADR 0003）**：把插件做成 huan 一等扩展机制。**(1) 形态**：顶层 `internal/plugin/` host + 领域包 capability 接口（首期只 `Deployer`）。**(2) Plugin 基接口极简**：只有 `Name()`；构造器吃配置，无 Init/Start/Stop；生命周期下沉到具体 capability。**(3) Capability 分散在领域包**：`internal/deploy/types.go::Deployer` 嵌入 `plugin.Plugin`；未来 `internal/payment/types.go::PaymentProvider` 同模式；`internal/plugin/` 不掺杂 domain 知识。**(4) 配置统一**：yaml 顶层 `plugins:<name>.*`；plugin 身份 = yaml key。**(5) 凭证**：`${VAR}` strict 插值（unset 报错，整个 huan.yaml 都支持）；无知名 env fallback。**(6) CLI**：per-capability verb（`huan deploy <name>` / 未来 `huan payment <name>`）+ 统一管理命令（`huan plugin list/info`）。**(7) 注册**：编译期 hardcoded 在 `cmd/huan/plugins.go`（composition root），断开循环 import；未知 plugin fail-fast。**未来加插件** = 新建领域包 + `cmd/huan/plugins.go` switch case + yaml 加 `plugins.<name>.*`，**不动 `internal/plugin/`**。备选方案 16 项否决理由见 ADR 0003。**首 PR（Cloudflare Pages）已实施**：`internal/plugin/plugin.go`（Plugin/Registry/Find[T]）+ `internal/config/interpolate.go`（${VAR} strict）+ `internal/deploy/{types,logging}.go`（Deployer + JSON logger）+ `internal/deploy/cloudflare/{hash,manifest,retry,client,pages,options,plugin,git}.go`（blake3 + 5-endpoint 协议 + JWT 双 token + retry 分类）+ `cmd/huan/{plugins,deploy,plugin_cmd}.go`（composition root + CLI 三层）+ `internal/deploy/cloudflare/integration/pages_test.go`（build tag integration）。**106 个测试全 PASS（含 race 检测）**。
- **Plugin 系统架构的 4 个关键 fork（grill-me 二轮收敛，2026-06-13 晚）**：(a) Plugin 接口「极简 + 构造器模式」否决「带 Init/Start/Stop」—— Go 习惯是构造器吃配置，两段式 Init dance 是 ceremony；(b) Registry 接线在 `cmd/huan/plugins.go` 而非 `internal/plugin/builtin.go` —— 避免循环 import（cloudflare→plugin→cloudflare 成环）；(c) yaml 用 `plugins:<name>.*` 而非 per-domain 顶层 key（`deploy:` / `payment:`） —— 否则 host 必须知道每个 domain 名，碎片化；(d) 注册用编译期 hardcoded 而非 `init()` 自注册 —— Go 反模式，init() 跨包字母序不可预测。**含义**：未来 grill-me 推 plugin 系统类架构时，可继续按这 4 个 fork 给推荐。
- **Cloudflare Pages API 实际协议（2026-06-13 晚实施前 Plan agent 反向工程 wrangler 源码得出，ADR 0002 §0/§7 修订）**：(a) **hash 算法是 blake3 不是 SHA256**：`hash = blake3(base64(content) + ext).hex()[:32]`，ext 是不带前导 dot 的扩展名。Go 库 `github.com/zeebo/blake3`。**含义**：任何 byte 错误整个 dedup 失效；hash 函数必须有 wrangler/python 外部参照向量测试。(b) **5 步协议不是 4 步**：token → check-missing → upload → **upsert-hashes** → deployment。upsert-hashes 告诉 CF 本次部署完整 hash 列表（不只是新传的）。(c) **双 token 模型**：`apiToken` 长期用于 upload-token GET + deployment POST；`jwt` 300s 短期用于 assets/* 端点（注意路径不带 `/accounts/{id}` 前缀，JWT 已 scoped）。(d) **HTTP POST 并行硬上限 3**（与 wrangler 一致），与 `--concurrency`（CPU 并行）解耦。(e) **资源硬上限**：单文件 25 MiB / 单 bucket 40 MiB / 单 bucket 2000 文件 / 总 20000 文件，manifest 阶段即检查。(f) **manifest key 必须前导 `/`**：`["/"+path]: file.hash`。**含义**：任何「按文档实施 API 协议」前必须先验证协议（参考实现 > 文档）。
- **Plugin 系统 4 个 Go 编译器/语法坑（2026-06-13 晚实施撞墙）**：(a) **`goto` 不能跳进 block** —— Go 编译器拒绝 `goto unsupported` 跳进 if 块；改成显式 helper 函数（如 `hasUnsupportedTarget()`）。(b) **回引号字符串不能内嵌双引号** —— `len(`name=""`)` parser 会困惑；用 `"name=\""` 转义或 `strings.Index(header, "name=\"")`。(c) **`*testing.T` 在 http.Handler 闭包里需手动注入** —— mockServer.handle 用 `t.Errorf` 必须把 `t` 作为 struct 字段传入（不能从外层闭包捕获，因为 handler 注册时机比 t 早）。(d) **`backoffSchedule` 必须是 var 不是 const** —— 测试需要临时覆盖时间常量，const 无法被覆盖；用 `t.Cleanup(func() { backoffSchedule = original })` 模式。**含义**：未来写带时间间隔/常量的代码，默认 var 不是 const，方便测试。
- **R2 sync 实施细节（PR2，2026-06-13 晚）**：(a) **minio.New 要 bare hostname** —— 传 `https://<acct>.r2.cloudflarestorage.com` 报「Endpoint url cannot have fully qualified paths」。正确格式是 bare hostname，scheme 由 `Options.Secure=true` 控制。(b) **抽象 r2ObjectClient 接口** —— 不要在测试里 mock 完整 S3 wire protocol（签名/重试/分块上传）。抽象出 `BucketExists/ListObjects/PutObject/RemoveObject` 接口，让 minioClient 实现生产、mockR2Client 实现测试。(c) **R2 etag = MD5**（与 Pages 的 blake3 完全不同）—— R2/S3 单 part 上传的 etag 是 hex(MD5(content))。本地算 MD5 对比即可。(d) **collection-not-interruption 并发上传要 mutex** —— 3 个 goroutine 并发上传，每个 uploadOne 失败时 append 到 `result.Failures` 必须持锁，否则 race。`go test -race` 抓住这个 bug。
- **Worker modules upload 实施细节（PR3，2026-06-13 晚）**：(a) **CF Workers multipart 必须有 filename header** —— script part 的 Content-Disposition 必须是 `form-data; name="<basename>"; filename="<basename>"`，缺 filename CF 拒绝。原 PostForm 只支持 name，必须扩展 formField struct 加 filename 字段 + 加 PutForm 方法。(b) **binding 类型映射 CF 期望的 JSON 字段名**：r2_bucket → `bucket`；kv_namespace → `namespace_id`（snake_case）；d1 → `id`；vars/secret_text → `text`（不是 value）。用户 yaml 用 camelCase，代码集中映射。(c) **route 只填 `zone_name`，不填 `zone`**（审计 H5 修正）—— wrangler 实证只填 `zone_name`；同时填两边可能被 CF 当 ambiguous 拒绝。(d) **跨 PR 的「未实施」断言会反向失效** —— PR2 写了 TestR2Sync_PluginDispatch_WorkerTarget 断言「PR3 not yet」错误。PR3 实施后必须主动改回。**含义**：跨 PR stub 测试要主动维护，PR 实施后立刻 grep 找出反向失效的断言。
- **审计修复关键决策（2026-06-13 晚，8 commits）**：(a) **concurrency 模型：CPU knob 暴露 + HTTP knob 不暴露** —— `opts.Concurrency` 管 CPU（hash/walk），HTTP=3 内部硬编码 + 任意 5xx 自动降到 1（不可逆）；Limiter 用 dual-semaphore + atomic.Bool 实现。(b) **审计 agent 报告必须实证复核** —— C4「wrangler 分片 check-missing」与 H6「binding 字段污染」都是推测，grep wrangler 源码后撤回。(c) **代码已正确时只补 observability + docs** —— H3 prune scope 代码本身正确（listAll 只列 configured prefixes），加 per-delete 结构化日志 + ADR §6.1 文档化用户责任即可。(d) **构造器绝不修改全局状态** —— `log.SetOutput(io.Discard)` 在 NewR2Syncer 里吃掉整个进程的 stdlib log；删掉，minio 警告默认显示。(e) **trace ID 必须在 early-return 前生成** —— 导出 `deploy.NewTraceID()`，plugin.Deploy 入口生成，传给 logger 和所有 Report 值。(f) **Report 字段语义严格** —— dryRunReport 不再用 `Skipped=len(assets)`（误导：dry-run 没查过 remote）；改 Attempted-only + 其它字段 0。(g) **dryRunReport 必须显式 0 字段** —— Attempted 表达「要处理的文件数」，Skipped 表达「remote 已有匹配 hash」（dry-run 不知道），失败=0，成功=0。

- **`huan release` 实施撞墙的 4 个坑（2026-06-13 夜）**：(a) **BSD sed 不支持 `\b` word boundary** —— `s/deploy\.NewLogger\b/.../g` 在 macOS 上 `\b` 被当作字面量，pattern 匹配不到任何东西，导致 `deploy.NewLogger(traceID)` 漏改。改用 `deploy.NewLogger(` （带左括号）作为 disambiguator。**含义**：跨平台 sed 用 `\b` 风险高，要么用 GNU sed（gmalloc 版本）要么用其他边界 trick。(b) **//go:embed VERSION 返回带尾随 `\n` 的字符串** —— VERSION 文件用文本编辑器存默认带 `\n`，`fmt.Printf("huan %s\n", ver)` 输出 "huan 0.1.0\n\n"。修：`init()` 内 `strings.TrimSpace(versionString)`。**含义**：任何 `//go:embed` 文本资源都要在 init() 或访问点 trim whitespace。(c) **gofmt 报告"未修改的文件"是预存在违规** —— manifest.go/options.go 在 gofmt -l 输出里出现但不是我改的，是仓库里早有的格式偏差。修：只对自己 touched 的文件跑 gofmt -w。**含义**：gofmt -l 在不干净仓库上输出 ≠ 自己造成的违规，需要用 git diff/filter。(d) **atomicWrite 必须用同目录的 temp 文件 + rename** —— 跨文件系统 rename 失败，必须保证 temp 与 final 在同一 mount 点；用 `os.CreateTemp(dir, ".tmp-*")`（dir 是 final 的目录）即可。**含义**：原子写文件的标准模式是 sibling temp + rename，不能用 `/tmp` 跨文件系统。
- **`go run` 不总是嵌入 VCS info（2026-06-13 夜 v0.1.0 发版时发现）**：Q14 把 `go run ./cmd/huan release` 定为规范调用，但 `go run` 在 clean working tree 上构建时 `debug.ReadBuildInfo()` 的 vcs.* 字段经常是空的（dirty tree 反而更可靠嵌入）。这导致 release manifest 的 git_sha 字段缺失。**修**：release.go 改为 shell out `git rev-parse --short=7 HEAD` + `git status --porcelain`，描述源码目录的实际 git 状态而非 operator binary 的构建状态——manifest 本来就该描述前者。**含义**：(a) 不要依赖 `debug.ReadBuildInfo()` 做 provenance，shell out git 才可靠；(b) "operator binary 嵌入的 VCS"和"源码目录的 git 状态"是两个不同概念，工具链产物（go run vs go build）会影响前者，shell out 永远拿后者；(c) 发现这个 bug 的契机是 v0.1.0 真发版时 manifest 缺 git_sha——任何发版工具必须在打 tag 前实测一次完整流程，单元测试 + dry-run 都发现不了这个 bug。

## 经验教训

- **huan build 非确定性**（2026-06-12 发现）：连续两次跑相同代码的 `./huan build`，输出有约 75 个文件不同（占 3.7%）。来源未确认——可能涉及 map iteration、文件扫描顺序、或时间戳。**含义**：单次 diff-build.sh 的 same/diff 计数有 ±75 噪声。判断"改动让 diff 增/减 N 个"时，N 在 75 以内**不可信**——必须用同一份 huan-baseline 跟新 huan-output 直接对比，而不是依赖绝对数字。
- **Go template + Scratch 引用语义 bug**（2026-06-12 撞墙，未解）：模板 `{{ $x := sort ($scratch.Get "key") }}{{ range $x }}` 中，`range $x` 实际遍历的是 `$scratch.data["key"]` 的**原始 slice**，不是 sort 返回值。证据：把 sort 改成 in-place mutate input slice 也不能改变 HTML 输出。试图给 `internal/template/funcs.go` 的 `sortFunc` 加 PageSlice/[]interface{} mutation 都失败。**含义**：补 Hugo 兼容排序的"在 sortFunc 里动手脚"路径走不通。下次尝试应该从 **Scratch.Add/Set 的 slice 共享** 或 **Go template 变量赋值在 pipe + variadic + 多返回值场景下的实际行为** 入手，并配合最小复现单元测试。
- **grill 实证必须 fresh build**（2026-06-12 教训）：早期基于 `/tmp/huan-output`（几天前的旧 build）做实证，误判 huan 有"effective-constraints 缺 15 章"的 bug。重新 build 后该 bug 消失——是旧输出造成的假象。**含义**：任何关于 huan 输出的实证，必须**当场重新跑** `./huan build`，不能信任 `/tmp` 里已有的输出。
- **CURRENT_STATE.md 的「5 类差异」描述部分过期**（2026-06-12 更新）：第 2 类「Hugo date 相同时顺序不稳定」是错的——实证发现 Hugo 有稳定 tiebreaker（推断为 `Date desc → lower(Title) asc → RelPath asc`）。stage 1 收尾后所有 5 类差异已按三维度尺子重新归类，详见 [`docs/standards/equivalence.md`](../docs/standards/equivalence.md) §4 与 §5。
- **Hugo CJK WordCount 的真实算法**（2026-06-12，Phase 3 确认）：不是 `unicode.Is(unicode.Han, r)` 之类的 Unicode 属性分类，而是 `strings.Fields(plain)` + 每个 field 若 `len==runeCount` 算 1 词（纯 ASCII），否则算 `runeCount`（含 CJK）。源码在 `hugolib/page__content.go`（Hugo master）。**含义**：CJK 段落里没有空格分隔的标点也按 rune 计入（如 `"巧合"` 算 4 词，不是 2 词）。huan 的实现见 `internal/build/summary.go::CountWordsInPlain`。
- **`div` 模板函数只支持 int 的 bug**（2026-06-12 Phase 3 暴露，Phase 3.5 已修复）：原 `div`/`add`/`sub`/`mul` 是 `func(int, int) int`，无法处理 zhurongshuo 模板 `{{ div $totalWords 10000.0 }}` 的 float64 字面量。**含义**：Phase 3.5 已通过 `toFloat64` coercion helper 修复（`internal/template/funcs.go`）。Stage 1 范畴内类似隐藏 bug 仍需在 stage 2 kickoff 时检查（例如其他模板函数签名假设）。
- **Hugo summary 用块边界，不是定长截断**（2026-06-12，Phase 5.5 确认）：`summaryLength=120` 是**下限**。Hugo 找到第 N 词位置后，**forward-scan 到包含该词的 block 元素的 close tag**（`</p>` / `</h1>`~`</h6>` / `</li>` / `</blockquote>` 等）才截断。所以长文章 summary 通常**远多于 N 词**——是包含第 N 词的整个第一/二段。GitHub issue #11863 是入口。huan 的实现是 `internal/build/summary.go::TruncateHTMLToBlockBoundary`（包装 `TruncateHTMLByWords` + `commonPrefixLen` trick + 26 种 block close tag forward-scan）。
- **`commonPrefixLen` trick**（2026-06-12，Phase 5.5 总结）：要在不破坏已有函数接口的前提下，拿到「带合成 close tag 的截断结果」在原输入中的真实切点 byte offset——比较原输入与截断结果的最长公共前缀即可。第一个分歧 byte 就是切点。比让原函数暴露内部状态干净。
- **stage 1 收尾的「3 项遗留」全部是误判**（2026-06-12 grill-me 复核）：原报告说 meta description 换行、RSS items 数量差、lastBuildDate 格式差——grill-me 全量复核后发现：(a) meta description 方向反了（huan 多行、Hugo plainify 折叠，根因是 `plainify` 函数没调 `collapseWhitespace`），(b) RSS items 数量差是 grep 命令误用（minified 单行 RSS 用 `grep -c "<item>"` 数行不数 occurrence），(c) lastBuildDate 两边 byte-identical。**含义**：发布"已完成"前必须用全量分析（不依赖 sample）+ 多角度 grep 验证；"3 项遗留"实际是 5 类真实差异（meta plainify / RSS URL 编码 / books part 顺序 / body 渲染细节 / minify artifacts），详见 `docs/progress/CURRENT_STATE.md` Stage 2 候选清单。
- **plainify 修复"两次撞墙"教训**（2026-06-12 stage 2 phase 1）：第一轮按"折叠 `\n` 为空格"方向修（collapseWhitespace + TrimSpace），实证发现差异增加 300 个文件——方向反了。Hugo `tpl/template.go:StripHTML` 实际用 placeholder 保留 `</p>` / `<br>` 边界为 `\n`，只对源 `\n` 与连续 whitespace 去重。**含义**：(a) Port 上游算法前必须读真实源码，不能凭"算法应该是 X"的直觉；(b) byte-level 实证（`od -c` 或 Python repr）比 grep 更可靠——前者能看到 `\n` / 空格 / 前导/尾随，后者会因 minified 单行 / regex 转义等假象误导；(c) 第一轮修复 reset 后用 Hugo 源码 Port 完整 StripHTML 算法，diff-build.sh 4 模式全部下降（seo 983 → 699，ai 323 → 36）。
- **stage 1 收尾「3 项遗留」+ stage 2 phase 2「RSS URL 编码」连续误判**（2026-06-12 两次全量复核教训）：stage 1 收尾报告的 3 项遗留经第一次全量复核发现全误判（meta 方向反 + RSS items 数量差不存在 + lastBuildDate 不存在）；第二次全量复核（stage 2 phase 2 启动前）发现 stage 2 候选 #2「RSS URL 编码 464 文件」也是误判（实际 0 文件有单纯 URL 编码差，真实分类是 187 个 RSS items 顺序差 + 17 个 items 内容差）。**含义**：(a) "影响 N 文件" 必须基于全量分析（不是 sample），否则会因 sample 选择偏差严重高估/低估；(b) byte-level 实证（`od -c` / Python repr）比 grep 在 `\n` / 空格 / 编码差异上更可靠；(c) 同一根因（中文排序）可能在不同表面（list 顺序 / RSS items 顺序 / books part 顺序）出现，归类时需要看根因不看表象。
- **taxonomy.Build 接收已排序 pages 的契约**（2026-06-13 stage 2 phase 2 教训）：`internal/taxonomy/taxonomy.go::Build` 不重排 term 内 pages（按 input order append），所以**调用方**必须传已排序的 pages。`internal/build/build.go` 用 `taxonomy.BuildAll(site.RegularPages)`（在 `content.BuildTree` 内部经 `sortPagesDefault` 排序），不要传 `pages`（目录扫描序，未排序）。否则 tags RSS items 顺序会按扫描序乱排。该契约由 `TestBuild_PreservesInputPageOrder` 守护。**含义**：未来若新增调用 `taxonomy.Build` 的地方，必须传 sorted pages。同理其他"按页排序输出"的子系统（list / section / RSS），都应消费 `site.RegularPages` 而不是原始 `pages`。
- **Hugo 用 collate 库做 locale-aware 排序**（2026-06-13 stage 2 phase 2 发现）：Hugo `resources/page/pages_sort.go:DefaultPageSort` 在 Date 相同时用 `langs.GetCollator1(currentSite.Language())` 构建 `golang.org/x/text/collate.Collator`，按 site language 的 CLDR 表排序（zh-cn = 拼音序）。huan 原用 `strings.ToLower(Title)` 字节级 UTF-8 比较，对中文章节标题完全错序（一 < 七 < 三 < 二 < 五 vs Hugo 二 < 六 < 七 < 三 < 四 < 五 < 一）。**含义**：(a) Port 上游排序算法必须查 collator 使用，不能假设字节级比较；(b) Go 标准库 `strings.Compare` 是 byte-level，要 locale-aware 排序必须用 `golang.org/x/text/collate`；(c) DefaultPageSort 完整链含 5+ 层 tiebreaker（Ordinal / Weight0 / Weight / Date / Collator LinkTitle / Path），不能只 Port 单层；(d) **排序修复要覆盖所有调用路径**——phase 2 初版只修了 sortPagesDefault，遗漏了 taxonomy.BuildAll 的 pages 参数（build.go:173），导致 154 个 tags RSS 文件未对齐，必须实证全量验证。
- **stage 2 phase 3 拆 3 子项 + 新发现候选**（2026-06-13）：原 stage 2 候选 #4「RSS items 内容差（17 文件）」经实施发现是 3 类独立问题：(a) hidden 字段未过滤（cascade inheritance 缺失），(b) posts section 缺 _index.md 时 auto-create 时机晚 + section.RegularPages 只含直接子，(c) BuildTaxonomyContext 错用 siteCtx.RegularPages 应为 term stubs。修复后发现**新的差异**：(d) 单个 term RSS（tags/{cjk}/）的 link/guid 应 percent-encode CJK（这是 stage 1 收尾报告里"RSS URL 编码 464 文件"的真正根因——之前误判不存在）；(e) 空 tag RSS 文件未生成（huan 缺 22 个 .xml）。**含义**：(a) "影响 N 文件"的估算必须基于全量分类（按差异类型，不按文件数），同根因的差异可能在不同表面出现；(b) 一次 phase 修复可能暴露下一层差异，需要递归调查；(c) "误判"和"实际不存在"之间需要严谨的 byte-level 验证才能区分。
- **Hugo taxonomy key 集合 vs term RegularPages 过滤是两个独立层**（2026-06-13 stage 2 phase 4）：Hugo 把**所有声明过该 tag 的非草稿页面**（含 hidden/`build.list: never`）作为 taxonomy key 来源——所以 hidden-only tag 仍出现为 key，Hugo 仍生成 `tags/{tag}/index.{html,xml}`；但 term page 的 `.RegularPages`（用于 RSS items 与 HTML 列表）会**排除** hidden pages。huan 之前把 `taxonomy.BuildAll(site.RegularPages)` 传成 filtered pages，导致 hidden-only tag 直接消失。**正确做法**：`taxonomy.BuildAll(site.Pages)`（不过滤），再在 `BuildTermContext` 内按 `c.Build.List == "never"` 过滤 page list。**含义**：Hugo 的过滤层级（key 集合 vs term 的 RegularPages）必须分开 port，不能合并；`site.Pages` vs `site.RegularPages` 的语义差就是这两层的边界。
- **Hugo term page `.Title` 用原始大小写，URL/路径用 urlize 后小写**（2026-06-13 stage 2 phase 4）：源 frontmatter 写 `tags: ["FANFAN"]`，Hugo term page 渲染 `<title>FANFAN on ...</title>`、`window.GA_PAGE_CONTEXT.title:"FANFAN"`，但 URL 是 `/tags/fanfan/`、文件路径是 `tags/fanfan/index.xml`。huan 之前两边都用 urlized key（小写）。**正确做法**：taxonomy 包新增 `BuildWithOriginalCase` 返回 `map[urlized_key]originalcased_name`（首见大小写优先），存到 `site.TaxonomyOriginalCase`，`BuildTermContext` 从中取原始大小写作为 Title。**含义**：taxonomy 的 "key" 和 "display name" 必须分开维护；CJK 大小写不敏感（CJK 不分大小写），但 ASCII tag 受影响。
- **Hugo 空 tag RSS 输出 `<lastBuildDate/>`**（2026-06-13 stage 2 phase 4）：Hugo 模板 `(index $pages.ByLastmod.Reverse 0).Lastmod.Format` 在 `$pages` 为空时返回 ""（Hugo nil-safe 模板引擎 + missingkey=zero），整体压缩为 `<lastBuildDate/>`。huan 的 `.First()` 返回 `&Context{}`（Lastmod 零值），`.Format` 输出 `Mon, 01 Jan 0001 00:00:00 +0000`。**正确做法**：新增 `RssLastBuildDate(ctx)` 模板函数，显式判断 `len(RegularPages)==0` 返回 ""，否则返回最新 Lastmod 格式化结果；rss.xml 模板改用 `{{ rssLastBuildDate . | safeHTML }}`。**含义**：Go template 的 nil-safe 与 Hugo 的 nil-safe 行为不同；当某个 Hugo 模板表达式涉及空集合时，huan 需要显式函数处理，不能依赖 Go template 默认行为。
- **Hugo term page 的 `.Section` = "tags"**（2026-06-13 stage 2 phase 4）：埋点脚本 `window.GA_PAGE_CONTEXT.section` 暴露——huan term page 之前 Section 为空字符串，Hugo 是 "tags"。修复：`BuildTermContext` 显式设置 `Section: "tags"`。**含义**：`tmpl.Context.Section` 字段在合成 context（term / taxonomy）中需要显式设置，不能默认空。
- **CJK URL 处理双轨：filesystem 用 CJK，URL 用 percent-encoded**（2026-06-13 stage 2 phase 4）：Hugo 对 CJK tag 的处理分两套——文件系统路径保留 CJK（`tags/专注/index.xml`），但所有 URL 输出（RSS link/guid、permalink、og:url、atom:link 等）必须 percent-encode（`tags/%E4%B8%93%E6%B3%A8/`）。huan 原 `URLEscape` 函数混用导致 term RSS link 错误保留 CJK。**含义**：(a) 任何处理 URL 的 helper 必须区分 filesystem path vs URL output 两个场景；(b) Hugo 的 urlize（filesystem）vs URL escaping（HTTP）是两个不同算法，不能混用；(c) phase 4 同时修了 4 个相邻 bug（lastBuildDate 空 / Section 字段 / SortDefault / 原始大小写保留）——这些都是 byte-match 的隐藏前提，单修主问题不够。
- **stage 2 phase 5d/e/f 五项根因各异**（2026-06-13）：同一次修复涵盖 5 个看似无关的差异：(a) WordCount 差 100 字实际是 footnote 渲染缺失 + 代码块 entity 编码差异；(b) list page part 顺序差实际是 `sort` 模板函数无 field 参数时不排序（缺 else 分支）；(c) GroupByDate 组内未排序；(d) `ExtractSummaryFromHTML` Port 漏 3 个 quirk（段尾 word 算法 / 中间 block tag / HTML-tag-shaped token）；(e) stateful HTML strip 处理 italic-at-start 边缘。**含义**：(a) 一个 visible diff 可能有多个底层根因，必须逐个调查；(b) Port 上游算法必须 byte-faithful——近似 Port 永远会有边界 case；(c) goldmark 缺 footnote extension 是隐性数据丢失，WordCount 差是症状不是根因；(d) sort/group 模板函数的 else 分支缺失是常见 bug 模式。
- **hugoSlugify 的 4 个边界行为**（2026-06-13 normalized diff 16→6）：(a) `_` 是 word char 不是 separator——Hugo 保留 `_`（如 `assignee_id` → `assignee_id`），huan 之前转为 `-`；(b) Unicode letter（π/M/Cyrillic 等）Hugo 保留并 lowercase——huan 之前全丢；(c) trailing dash 不 Trim——Hugo 保留 `413-管道与重定向-` 的尾部 `-`，huan 之前 `strings.Trim(..., "-")`；(d) 多行 heading（`\n` 在 blockquote 内 heading）Hugo 只取最后一段做 slug——huan 之前 `\n` 转 `-` 产生 `你-great-...` 前缀。**含义**：slugify 算法必须 byte-faithful port，每个字符的处理规则（word char vs separator vs drop）都影响 downstream。
- **emoji 短码需跳过 `<pre>/<code>` 区域**（2026-06-13）：Hugo 的 emoji 渲染（`:white_check_mark:` → `&#9989;`）只作用于 markdown 文本，不影响 code block 内容。huan 的后处理实现须用 protected regions 方案（先收集所有 `<pre>...</pre>` 和 `<code>...</code>` 的起止位置，再逐 match 判断是否在 protected 内），不能逐字符扫描遇到 `<code>` 就 skip——因为 `<pre>` 内嵌套的 `<code>` 会导致 skip 过早结束。**含义**：HTML 标签嵌套场景下的后处理必须用 range-based 过滤，不能靠顺序扫描。
- **CI build 时 huan.yaml 的 strict `${VAR}` 插值要求所有引用的 env var 都设置**（2026-06-14 zhurongshuo 上线后核实）：zhurongshuo CI `.github/workflows/deploy.yml` 跑 `huan build` 时，因为 huan.yaml 引用了 `CLOUDFLARE_ACCOUNT_ID` / `CLOUDFLARE_API_TOKEN` / `CLOUDFLARE_R2_ACCESS_KEY_ID` / `CLOUDFLARE_R2_SECRET_ACCESS_KEY` 等 4 个 deploy 凭证变量（用于 `huan deploy cloudflare`），strict 插值要求全部设置——即使 build 本身不需要这些。zhurongshuo 的解法是在 build step 加 dummy env vars（值 = `dummy`）。**含义**：(a) strict 插值的 fail-fast 是正确的——避免 typo / 缺凭证静默走默认值；(b) 但 deploy plugin 的凭证在 build 阶段也会被解析，是个意外的耦合；(c) 未来若要让 build 不需要 deploy 凭证，可能要把 yaml 解析 lazy 化（按需读 `plugins.<name>.*`），或拆分 build config 和 deploy config；(d) 当前 dummy 法是合理的 workaround，CI 文档里已 inline comment 解释。
- **Hugo→huan 迁移的"混合部署"模式**（2026-06-14 zhurongshuo 实证）：HTML 走 GitHub Pages 链路（push master → CI `huan build` → GH Pages → Cloudflare 自动镜像做 CDN），**不走** huan 的 Cloudflare Pages 直连插件。但 Images 和 Worker 直接用 huan 的 Cloudflare deploy 插件（`huan deploy cloudflare r2` / `huan deploy cloudflare worker`），本地 `deploy.sh` 协调三路。**含义**：(a) Cloudflare 插件不必"全有或全无"，可以分资源类型选择性使用；(b) HTML 走 GH Pages 中转保留了 GitHub-based 回滚能力（revert commit 即可），但牺牲了 Cloudflare Pages 的 preview deployment / atomic deploy 能力；(c) 本地 `deploy.sh` 是"协调器"而非"构建器"——构建已经 100% 转到 CI。
